## 三、可扩展性评估

### 3.1 页面 UI 复用性

**现状**：
- smartled 与 socket 各自实现，结构相似但未抽象
- 共用 `custom-header` 组件
- 未抽出统一的设备卡片、控制卡片等通用组件

**改进建议**：
- 抽离 `device-card` 组件：设备图标、名称、连接状态
- 抽离 `control-card` 组件：通用控制区域（标题 + 内容）
- 新建设备页时，以 smartled 或 socket 为模板复制并改业务逻辑即可

### 3.2 阿里云连接模块

**现状**：
- `device-service.js`：设备服务，封装 `getDeviceStatus`、`controlDevice`、`controlPump`、`toggleSmartPlug`、`setLightBrightness` 等
- `aliyun-connector.js`：底层连接，支持 `queryDeviceData`、`controlDevice`、`queryDeviceStatus`
- `iot-config.js`：集中配置

**优点**：
- 分层清晰，device-service 面向业务
- 配置集中在 iot-config

**不足（部分已演进）**：
- ~~连接器 Identifier 写死~~：已支持智能灯默认 **`SetDeviceProperty`**、可配置 `smartledUseInvokeThingService` / `socketThingServiceIdentifier`；`invokeService` 可自定义 Identifier
- 新增设备类型仍需在 device-service 或页面层扩展，缺少完全统一的「配置驱动」入口

**改进建议**：
1. 继续按产品维护 `iot-config` 与 TSL 文档（见 `docs/tsl/`、`阿里云IoT连接配置指南.md`）
2. 新设备优先复用 `invokeService` / `connector.setDeviceProperty`
3. （可选）在 iot-config 中按设备类型聚合配置（如 `deviceTypes`）

### 3.3 新增设备页面流程（建议）

1. 在 `config/iot-config.js` 增加 `deviceNameXXX`
2. 在 `app.json` 增加新页面路径
3. 在 `pages/index` 的 devices 中增加入口
4. 复制 `pages/smartled` 或 `pages/socket` 作为模板，修改：
   - 设备配置读取
   - 调用 `deviceService.getDeviceStatus(deviceId)`、`deviceService.controlDevice(deviceId, cmd)` 等
   - 页面 UI 与业务逻辑
